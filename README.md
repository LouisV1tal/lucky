# Система учёта заказов ковров (MVP)

Реализация MVP-скоупа из технического задания: клиенты, заказы с товарами,
QR-код заказа, печать бирок/накладной (PDF), производственная цепочка статусов
с переходом «на следующий этап» одним нажатием, история статусов, роли
(администратор / оператор / производство / курьер), Telegram-бот для клиентов
(привязка номера, просмотр статуса, автоуведомление «Готов к доставке»),
экспорт реестра в Excel.

Что **не** реализовано в этой версии (см. раздел 14 ТЗ "Функции для второй
версии") и оставлено как TODO: расчёт стоимости/оплата, подпись клиента на
экране, SMS-канал, печать в ZPL для промышленных термопринтеров, полноценные
аналитические отчёты, статусы на уровне отдельного товара, полноценный
офлайн-режим с конфликт-резолвером (заложена архитектурная возможность, но
сама синхронизация не реализована).

## Стек

- **Backend**: Node.js, Express, Prisma ORM, PostgreSQL
- **Frontend**: React (Vite), обычный REST через axios, адаптировано под планшет
- **Telegram-бот**: Node.js, Telegraf
- **Инфраструктура**: Docker Compose

## Быстрый запуск локально

Требуется Docker и Docker Compose.

```bash
cp .env.example .env
# откройте .env и подставьте:
#   - JWT_SECRET — любую длинную случайную строку
#   - BOT_TOKEN — токен вашего Telegram-бота от @BotFather (можно оставить пустым для теста без бота)

docker-compose up --build
```

После запуска:
- Frontend (веб-панель): http://localhost:5173
- Backend API: http://localhost:4000
- PostgreSQL: localhost:5432

При первом запуске backend автоматически применит схему БД (`prisma db push`)
и создаст демо-пользователей:

| Логин | Пароль | Роль |
|---|---|---|
| admin | admin123 | Администратор |
| operator1 | operator123 | Оператор |
| production1 | production123 | Сотрудник производства |
| courier1 | courier123 | Курьер |

**Обязательно смените эти пароли перед реальным использованием** — это
демо-доступы для проверки функциональности.

## Настройка Telegram-бота

1. Напишите [@BotFather](https://t.me/BotFather) в Telegram, создайте бота командой `/newbot`, скопируйте токен.
2. Вставьте токен в `.env` как `BOT_TOKEN`.
3. Перезапустите `docker-compose up --build telegram-bot`.
4. Клиент пишет боту `/start`, отправляет номер телефона (должен совпадать с номером в системе) → видит свои заказы командой `/my_orders`.

## Важные технические примечания

- **QR-код**: генерируется как непредсказуемый токен (UUID v4), не содержит
  персональных данных. По умолчанию ссылка вида `https://orders.example.com/o/{token}`
  — задайте свой домен через переменную `PUBLIC_ORDER_URL` в окружении backend.
- **Сканирование**: страница `/scan` во фронтенде принимает ввод как от
  USB/Bluetooth-сканера штрихкодов (они эмулируют клавиатуру), так и вручную.
  Полноценное сканирование через камеру браузера требует HTTPS и отдельного
  тестирования на конкретной модели планшета — это сознательно оставлено на
  следующий этап (см. вопрос №6 в разделе вопросов к заказчику исходного ТЗ).
- **Экспорт в Excel**: доступен администратору по кнопке/эндпоинту
  `GET /api/v1/export/orders.xlsx` с фильтрами.
- **Печать бирок**: формат 58×40 мм (стандарт термопринтера этикеток), PDF.
  Если у вас конкретная модель термопринтера, поддерживающая ZPL/EPL —
  печать по PDF всё равно будет работать через обычный принтер, но для
  прямой печати на термопринтер нужна отдельная доработка (см. ТЗ, риск №6).

## Безопасность перед продакшн-использованием

Обязательно сделайте до реального использования с настоящими данными клиентов:
1. Смените все демо-пароли и `JWT_SECRET`.
2. Включите HTTPS (например, через reverse-proxy типа Caddy/Traefik/Nginx + Let's Encrypt).
3. Ограничьте доступ к PostgreSQL (порт 5432) только для backend-контейнера, уберите публичный проброс порта в `docker-compose.yml` в продакшене.
4. Настройте регулярное резервное копирование volume `pgdata`.
5. Проверьте требования локального законодательства о персональных данных (см. вопрос №5 из исходного ТЗ) — это может повлиять на то, где физически может размещаться сервер.
